## 🤖 AI押题

### 1. 介绍下AI测试平台

我当时做的AI 智能测试管理平台，是一个集用例管理、接口自动化、UI 自动化、数据工厂 等一站式测试平台，我主要对平台进行AI 相关的接入优化， 比如 AI 分析需求 用例生成 添加lvm 图片识别，接入llm AI 用例平台环境，基于browser-use 的AI ui 自动化 等

### 2. AI Agent你是怎么做的

有做过的，其中 browser-use 的 UI自动化就是 aigent模式 基于 browser-use 库 + LangChain，核心是一个用户输入自然语言任务描述 执行 ui 自动化的一个 ai agnet，第一步 用户输入 的内容 用LLM 将内容拆解为一个个子步骤 第二部Agent 主循环 ① 获取当前浏览器状态 ② LLM 决策下一步操作（点击、输入、滚动、断言...）③ 调用工具 执行浏览器操作 ④ 回调 → 进度追踪 ⑤ 重复直到任务完成或超时

### 3. 具体AI Agent状态管理怎么做的

因为 browser-use 本身不提供持久化，所以我是通过自定义方法 和回调来跟踪的，LLM 自己在执行过程中上报状态，比如在 子任务完成，子任务失败，完成任务状态更新，全部任务完成，等在 Prompt 里强制要求 LLM 必须在每个子任务结束后调用这些标记动作：step 结束会触发回调，做三件事：

检查是否超时或被用户取消（should_stop）
解析 LLM 的 model_output，提取实际执行的操作和状态变化
通过 emit_callback 推送到前端，用户能看到实时进度

### 4. 开发中有没有什么难忘的问题

LLM 流式生成测试用例 + 截断续写的边界问题，用 SSE 流式推送到前端。LLM 有 token 上限，复杂需求经常生到一半被截断。比如生成了 15 个用例，第 15 个只写到一半就断了。续写时 LLM 忘了前面写了什么，可能重复生成或者格式不一致。试过传历史对话做上下文，但 token 消耗太大，后面写了两个方法为主要1. 断点修复：识别并修复最后一个不完整的用例 2. 续写时把前 3 个用例作为格式参考，但不全量传，最多续写 5 次，5 次后还超长就返回当前结果，不无限循环。

### 5. 大模型之间是怎么选型的

现在的文本用 deepseek v4 ，然后lvm 图片识别用的智普 主要是以 成本 中文理解 响应速度 等维度吧，用的一般情况比较轻任务用 v4-flash 上下文比较长的任务还是以v4-pro 效果可能会更好点，相对于成本也会减少

### 6. 怎么保证AI生成的用例质量

AI 生成不是一步到位，分步骤

先是提示词， prompt 里会明确要求 LLM 按固定格式输出：用例标题、前置条件、测试步骤、预期结果、优先级，缺任何一个字段都算无效用例。同时会列出必须覆盖的边界条件类型，比如数值型字段要测 0、负数、边界值、超长值。

然后是换一个角色（reviewer 角色）对生成的用例进行评审，检查格式完整性、边界覆盖率，等

最后根据 评审意见再去修改用例，两轮 LLM 交叉验证，能过滤掉大部分明显的幻觉和遗漏

③生成后校验机制：格式校验、去重合并、覆盖率分析

最后 生成出来的用例也不是直接可用的，而是用例状态变成待确认，人工确认完成用例才是一个可用的状态

### 7. AI生成的测试用例不可用怎么办

从 prompt 调优、模型切换、多轮生成三层去修，还不行就人工补充，把失败案例沉淀下来反哺 prompt。

**第一层，看 prompt 够不够具体。**很多时候不是 AI 不行，是 prompt 没写清楚。只说"给我生成登录页面的测试用例"肯定写得泛。要给出具体的字段定义、业务规则、必须覆盖的边界类型。prompt 模板会持续迭代，如果某类场景经常生成得不好就去改模板，加 few-shot 示例。

**第二层，换模型或者调参数。**有些模型适合创意类任务但不适合结构化输出。DeepSeek v4 长文本理解好，但如果发现生成的用例老漏字段，就切到 Qwen 试试，或者调低 temperature 让输出更确定。

**第三层，多轮自愈。**第一次生成质量不行，不走评审流程了，直接用更严格的约束重新生成。review 模型打分低的时候直接触发重新生成，而不是让人去改。多轮交叉验证兜底。

**第四层，人工兜底。**调了两三轮还不理想——需求太复杂或业务规则太特殊——直接人工写。AI 至少提供了初稿框架，比从零开始省时间。失败的案例会沉淀下来：什么需求、什么模型参数、哪些维度不合格，后续优化 prompt 时用得上。

核心态度：AI 是提效工具不是替代。帮我把 70% 常规场景覆盖了，剩下 30% 人工处理，整体效率还是大幅提升。

### 23. 模型调度和切换是怎么做的

设计了一个统一的AIModelConfig管理层，通过role区分场景。数据库里定义了writer、reviewer、vision、data_generator、browser_use_text等8种角色。

每个角色独立配置provider/api_key/base_url/model_name/temperature/top_p，PromptConfig支持版本管理。

调用方式两种——requirement_analysis和api_testing用httpx.AsyncClient直调OpenAI兼容API；ui_automation因为browser-use依赖走LangChain ChatOpenAI包装。

切换模型不用改代码，页面上把role的base_url和api_key切一下就行。

### 24. browser-use和Selenium比实际效果怎么样

Selenium执行稳定约98%，但对元素定位强依赖，前端改className就红，复杂场景半天到一天写脚本。

browser-use写prompt代替写脚本（几分钟），前端小改动不影响，但执行时间波动大，稳定性约85%。

我的策略是70%回归用Selenium（页面验证、数据校验），30%复杂E2E用browser-use（登录→搜索→下单→支付）。

踩坑：Kimi temperature强制1.0、标签页焦点丢失"幽灵点击"、Chrome僵尸进程堆积。

### 25. AI做失败根因分析怎么实现的

采集失败上下文（日志堆栈+接口报文+UI截图+console错误）→信息聚合为prompt→LLM判断是环境/业务/数据问题→历史匹配→报告输出。

初期准确率50%，优化prompt结构后稳定在70%左右。定位时间从15分钟缩短到3分钟。

短板是多服务链路追踪时单凭测试日志不够定位到具体微服务，需要接入APM数据。

### 26. 在Gate.io落地效果怎么样

在永续合约和余币宝两个业务落地。用例编写效率提升60%（4-5小时→1.5-2小时），AI生成可用率约70%。

回归测试全量2000+用例缩到AI推荐的800+，执行时间从2小时缩到45分钟，缺陷检出率保持95%以上。

线上漏测率下降30%，发布周期从2周缩短到1周。

### 27. 需求文档里有截图/流程图怎么处理

在用例生成管线中加预处理步骤——解析文档检测![image](url)引用→自动下载图片→调用vision角色的LVM模型转文本描述→插入到原位置再喂给writer模型。

比如登录页面截图，AI描述成"包含用户名输入框、密码输入框、登录按钮、验证码输入框"，writer模型基于此生成用例。

LVM选智谱，对比过通义千问视觉和GPT-4V，智谱在中文UI截图识别上最稳定。

### 29. AI测试平台后续还有什么改进方向

当前改进思路主要两个方向：

**第一，多Agent协作。**现在单Agent能力有限，质量天花板明显。后续想做多Agent协作——一个Agent理解需求、一个Agent生成用例、一个评审、一个执行、一个分析结果。它们互相对话和交叉验证，评审Agent发现问题直接打回修改，减少人工介入。

**第二，AI生产分析——从被动工具变成主动发现问题。**目前AI更多是等人工指令的被动角色。后续想接入生产环境运行日志，让AI主动分析线上流量特征：监控接口响应时间趋势、识别高频报错的参数组合、分析调用链数据。AI不再等人说"帮我查一下这个日志"，而是主动推送——"接口A的TP99最近三天上升50%，建议排查"或"接口B的参数组合X在线上频繁报错，测试未覆盖，建议补充用例"。从人用AI的工具，转变成AI主动产出分析、推动质量改进的驱动力。

## 🏢 业务测试

### 7. 搜索引擎/算法推荐的测试方法和策略是什么

搜索算法的测试我主要用这么几种方法：

第一，AB对比验证——搭建两套环境，老算法和新的算法并行，用同样的测试词集去跑，对比召回率和排序效果。

第二，测试词集构建——覆盖核心词、长尾词、品牌词、错别字、同义词的测试集，大概有500多条，用来系统化评估搜索质量。

第三，测试集的设计思路 多级兜底策略——核心词匹配→相似度→品牌词推荐→模糊搜索，每一级都要验证是否正确触发。

### 8. C端和B端测试从测试方法和策略上有什么区别

从测试策略上来说，我理解C端和B端的差异主要有这几个方面：

第一，测试重点不同——C端更关注用户体验、页面兼容性、弱网场景、中断测试、性能流畅度；B端更关注数据准确性、业务规则、权限控制、审批流程、批量操作。

第二，测试方法不同——C端需要做大量的兼容性测试（不同手机型号、操作系统版本）、网络专项测试；B端更注重数据对账、状态流转、异常回滚这些业务逻辑验证。

第三，质量要求不同——C端一个小的UI问题可能影响大量用户体验；B端一个数据错误可能导致财务损失，所以B端对数据准确性要求更高。

### 11. 库存管理包含哪些核心业务场景？比如盘点、补货怎么测

库存管理的核心场景我归纳为几个方面：

第一，库存的增减——采购入库增加库存，销售出库减少库存。

第二，库存不足时的自动补货机制——当销售订单触发的库存不足时，系统会自动创建采购询价单，这里要区分直发和非直发两种模式。

第三，库存盘点——定期盘点保证系统库存和实物库存一致。

第四，库存数据的准确性验证——确保不会出现超卖或者库存负数的情况。

第五，异常场景——比如库存锁定、库存占用、库存释放的逻辑。

### 13. 交易系统的测试策略怎么设计

从资金安全、核心链路、异常容错三个维度按P0/P1/P2分层。

P0资金安全——资产精度验证、账务三端一致性（数据库余额/缓存余额/用户流水日志）、幂等校验（重复下单不导致资产多次变更）、异常补偿（清算到一半服务重启后的补偿逻辑）。

P1核心业务——开仓→持仓→平仓完整链路、资金费率8小时结算快照准确性、标记价格计算（防价格操纵）、强平触发条件（30+边界场景）。

P2异常性能——高并发下单/撤单/强清的数据一致性、行情源断了的降级容错、越权操作（A操作B订单）。

每次发布P0全量自动过，P1按变更影响范围选测。

### 14. 永续合约的清算怎么测试

永续合约没有到期日，核心是测试资金费率对持仓成本的调节机制和极端行情下的强平清算链路。

资金费率——费率计算基于利率和溢价指数，每8小时结算一次快照。方向不能搞反。封顶逻辑±0.75%。

强平清算——保证金率=维持保证金率时触发。测试正常强平、阶梯强平（大仓位分批平）、穿仓（保险基金/ADL）、高并发多用户同时强平的排队机制。

我设计了一套对账脚本，强平完成后从数据库余额、缓存快照、用户流水三个来源逐笔比对，对不上标红。

### 15. 供应链ERP端到端链路怎么测

采购到付款涉及ERP/WMS/财务三个系统联调。

业务链路——采购下单（ERP）→供应商发货→WMS收货→入库单回传→财务校验→付款。

第一层，单据流转正确性——采购单审批流、入库单回传状态更新、财务自动触发付款。

第二层，数据一致性——ERP库存台账vs WMS实际库存每日对账、财务流水vs业务单据逐笔匹配。

第三层，异常场景——部分入库（分批到货）、退换货财务自动扣减、审批人离职审批流卡住。

我设计了E2E对账脚本，模拟完整采购链路，结束后自动比对各系统数据状态，字段对不上标红。

### 16. 大数据T+1流水线怎么保证数据不丢不重

源头计数→抽取计数→清洗计数→加载计数，逐层比对；唯一键去重检测；数据质量监控（空值/异常值/格式）；对账机制（源系统vs数仓vs目标系统三方交叉验证）；告警机制（行数波动超阈值自动告警）；数据回溯（重跑）方案验证，重跑后数据与第一次一致。

### 17. Odoo ERP的财务核算链路怎么测试

链路：采购订单→入库→发票→应付账款→付款→凭证→总账→财务报表。

测试：正向业务流打通；反向冲销/红冲；数据对账（业务系统vs财务系统）；金额精度/舍入；汇率转换；关账/开账验证。

### 18. Kafka在数据链路测试中关注哪些点

消息格式——字段定义、类型、是否兼容。Topic/Partition——分区策略、消息顺序。消费——offset提交方式、重复消费、消息积压、消费延迟。异常——网络中断/Kafka宕机后恢复、失败重试、幂等落库。监控——消费延迟告警、消息量趋势。

## 📊 测试管理

### 9. 需求频繁变更、提测延期，但上线时间不能变，如何保质保量

这种情况其实项目里很常见，我的处理思路是这样的：

第一步，风险前置——第一时间把延期的风险和影响范围反馈给上级和产品，让大家有心理预期。看一下能不能协调人员进入

第二步，按优先级分层——把测试用例按P0/P1/P2分层，P0的核心流程必须全部覆盖，P1的尽量覆盖，P2的跟产品确认是否可以放到下个版本。

第三步，提前介入——跟开发沟通能不能先把核心接口提测，我先做接口层的验证，等全部提测后再跑完整的业务流程。

第四步，如果是模块之间相对独立，可以分模块先测，最后集成验证。

第五步，如果确实时间不够，加班是最后的手段，但前提是前面几步都做了，确保加班的时间花在最有价值的地方。

### 19. 版本迭代快怎么保证质量不滑坡

核心是风险驱动测试，资源有限的情况下区分"必须测的"和"可以放一放的"。

第一步，风险评估定范围——跟开发确认变更影响范围，核心链路全覆盖，边缘冒烟。AI分析代码变更推荐回归用例，缩到40%。

第二步，自动化兜底——P0用例全自动化，CI门禁接口通过率<95%阻断发布。UI自动化走Selenium稳定回归。

第三步，人工聚焦高风险——新功能逻辑、资金安全、复杂业务流转手工测。常规页面用AI截图对比回归。

以前踩过坑——时间紧砍回归，线上漏几个bug花了两倍时间修。后来建了质量门禁，核心链路一票否决。

### 20. 怎么衡量测试质量？哪个指标最有用

最关注线上漏测率（目标≤5%），最不看重用例执行率和自动化覆盖率数字。

关键指标——线上漏测率（最真实反映有效性）、缺陷发现阶段分布（左移效果）、P0用例通过率（≥95%阻断发布）、回归集有效性（长期不发现bug说明需要更新）。

Vanity Metric——用例执行率（跑低价值用例100%也没意义）、自动化覆盖率数字（质量比数量重要）、Bug总数（一个资损bug比100个文案严重）。

### 21. 测试左移你做了什么

需求阶段——AI分析PRD自动生成测试要点，在需求评审会上直接提风险。比如"批量撤单并发性能"在需求阶段讨论清楚。

开发阶段——推动自测+CI门禁（代码检查/单元测试），提测问题减少30%。

用例设计——AI辅助生成测试用例，评审时发现需求盲区倒推补充。

线上监控反哺——线上异常自动上报，AI分析后补充用例库。

### 22. 上线后漏测了怎么复盘和改进

止损定位影响范围→根因分析（需求理解偏差/用例覆盖不足/环境数据差异/回归策略遗漏）→改进措施（补充用例到用例库、增加自动化覆盖、优化测试checklist、加强需求评审参与）→沉淀为质量指标。核心思路是找流程漏洞而非追责。

### 28. 怎么跟CI/CD集成实现质量左移

质量左移到需求阶段——AI分析PRD生成测试要点和风险提示，测试在需求评审会提前识别风险。
